## Working Rule

| 规则 | 说明 |
| --- | --- |
| 一次只认领一个 bug | 不要一口气并发修 5 个页面 |
| 先改状态再开工 | 在 `index.md` 里把 bug 改到 `in_progress` 并写 owner |
| 只按 bug 单边界修 | 不顺手扩成大重构 |
| 修完最多到 `fixed_pending_verify` | 不允许自己改成 `verified` 或 `closed` |
| 自测结果必须回写 bug 单 | 不接受“本地看起来好了” |

## Close Rule

| 状态 | 谁能改 |
| --- | --- |
| `in_progress` | Fixer |
| `fixed_pending_verify` | Fixer |
| `verified` | Verifier |
| `closed` | 总控线程 |

硬规则：

- Fixer 不能直接把自己修的 bug 标成 `verified`
- Fixer 不能直接把自己修的 bug 标成 `closed`
- 用户端 bug 先由测试猫独立复核推进到 `verified`，再由测试总监按 `Repro` 逐单真复测后才能进入 `closed`
- 管理后台 bug 先由测试狗独立复核推进到 `verified`，再由测试总监按 `Repro` 逐单真复测后才能进入 `closed`
- 跨域 bug 由总控线程按 `Repro` 逐单真复测并决定最终收口

## Bug Families

修复时先判断自己拿到的是哪一类：

| 类型 | 典型 bug | 修法提示 |
| --- | --- | --- |
| 吞错空态 | `025` `026` | 不要 `catch -> []`；改成显式错误态或重试 |
| 假成功 | `018` | 不要先本地成功再补回滚；失败要停在失败 |
| 假配置 UI | `013` `029` | 不能展示超出 payload 的配置项 |
| 假业务数据 | `014` `028` | 没有真实值就不要补“更好看”的默认业务数据 |
| 未闭环能力伪装可用 | `027` `028` `030` | 未承接时应明确“未开通/建设中”，不要继续假装可用 |
| 错路由 | `006` `031` | 文案和真实跳转要一致 |

## Fix Boundary

| 可以做 | 不要做 |
| --- | --- |
| 增加页面显式错误态 | 顺手把整个页面重写一遍 |
| 隐藏/禁用未接线按钮 | 自己补一整套后端新接口 |
| 把错误路由改正确 | 顺手重构整个导航体系 |
| 删除假数据/假配置 | 把相关页面大范围换视觉 |
| 让页面只暴露真实 payload 支持的能力 | 未经确认扩展 shared 契约和数据库链路 |

## Verifier Note

测试猫复核用户端时重点看：

- 是真修掉了，还是从报错改成了假空态
- 是真走到了真实路由，还是只是文案改了
- 是真删掉假数据了，还是换了另一组默认值
- 是真收敛到真实 payload 了，还是 UI 还在暗示用户能力存在

所以 Fixer 不要只改表面文案。

测试狗复核管理后台时重点看：

- 是真改到后台实际页面和后台接口，不是只改文案或局部假状态
- 是真按后台权限边界处理，不是借用户端接口或假数据兜底
- 是真修掉后台筛选、表格、分页、状态口径问题，不是本地伪造统计
- 是真把后台失败暴露出来，不是继续吞成空表、0 值或默认态

测试总监最终复测时重点看：

- 是否亲自按 bug 单 `Repro` 走到了真实页面或真实接口场景
- 看到的实际运行结果是否与 bug 单 `Expected` 一致
- 该单的复现路径和修复证据是否真的被完整覆盖
- 域验证方给出的 `verified` 结论是否足以支撑 `closed`
- 相邻页面、相邻接口、相邻角色有没有同根因残留
- 修复是否把“错误”换成了“假正常”或“假空态”
