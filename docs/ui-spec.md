# UI Spec

## 用途

这份文档不是设计稿，而是当前 `umi/` 工作区的 UI 约束文档。

目标：

- 减少线程凭感觉猜页面结构、文案、交互
- 明确哪些页面以老系统为参考
- 明确哪些改动可以做，哪些不能做

使用顺序：

1. 先看 [AGENTS.md](/Users/ezreal/Downloads/joy/umi/AGENTS.md)
2. 再看本文件
3. 再看具体页面对应的旧系统页面或当前页面代码

## 总原则

1. 不要凭页面现状猜产品要求。
2. 不要把旧系统实现细节直接等同于新系统最终方案。
3. 不要因为缺说明就自行重做信息层级、筛选器、按钮区或流程步骤。
4. 如果页面属于“迁移对齐型”，优先对齐旧系统；如果属于“新系统新增型”，优先对齐当前文档和已拍板结构。

## 事实源

UI 相关任务默认按这个顺序找事实源：

1. 本文件
2. [docs/flows.md](/Users/ezreal/Downloads/joy/umi/docs/flows.md)
3. 当前 `umi` 代码
4. 老系统页面

如果这四层都没有明确答案，再允许做最小假设。

## 页面类型

### 1. 迁移对齐型页面

这类页面以老系统为主要参考：

- 用户端页面
- 老后台已有的成熟列表页、详情页、配置页

约束：

- 先还原结构和交互，再谈优化
- 不要擅自新增模块
- 不要擅自合并信息区块
- 不要改按钮语义

### 2. 新系统新增型页面

这类页面没有完整旧页面可抄：

- 新后台权限体系
- 新配置层/结果层页面
- 新表驱动的管理页

约束：

- 先按数据结构设计页面
- 先保证字段和流程准确
- UI 可以简洁，但不能脑补复杂玩法

## 用户端规则

用户端默认以老系统 `frontend/` 为视觉和交互基准。

补充规则：

- 颜色、字号、圆角、卡片层级、底部导航，不要自行发明新体系
- 图标优先沿用旧系统风格
- 用户端若无明确新需求，不做“设计升级式重画”

直接参考：

- 旧前端目录：[/Users/ezreal/Downloads/joy/frontend](/Users/ezreal/Downloads/joy/frontend)
- 细则文档：[docs/ui-rules.md](/Users/ezreal/Downloads/joy/umi/docs/ui-rules.md)

## 管理端规则

管理端不要靠猜“后台大概应该长什么样”。

默认规则：

- 列表页优先：筛选区 + 表格 + 行操作
- 详情页优先：基础信息 + 状态信息 + 关联记录
- 配置页优先：配置表单 + 当前配置列表
- 结果页优先：时间/周期筛选 + 结果列表

如果是强数据驱动页面，先让字段、状态、过滤条件完整，再做视觉润色。

## Do Not Assume

- 不要默认页面缺一个模块就应该补一个模块
- 不要默认页面有个按钮就一定需要弹窗
- 不要默认老系统里的 mock 文案就是新系统最终文案
- 不要默认老系统字段名等于当前数据库字段名
- 不要默认后台页面能直接复用用户端组件结构

## 高风险猜测点

这些最容易被线程“脑补”，需要特别克制：

- 列表页筛选条件
- 状态标签文案
- 弹窗内容和确认动作
- 卡片统计项
- 空状态文案
- 页面顶部说明区
- 表单步骤数

遇到这些点，优先查 `flows.md`、`db.md`、`schema-reference.md`，不要直接猜。

## 输出标准

一个 UI 任务完成时，至少要能回答：

- 这个页面服务哪个业务对象
- 核心数据来自哪张表或哪条接口
- 页面是配置页、结果页、列表页还是详情页
- 哪些交互是必要的
- 哪些交互是当前明确不做的

如果这 5 个问题回答不清，说明页面定义还不够，不应该直接开做。
