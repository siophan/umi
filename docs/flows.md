# Flows

## 用途

这份文档用于约束“业务流程怎么理解”，避免线程从页面文案或局部代码倒推出错误流程。

它不替代数据库文档，也不替代 UI 文档。

分工：

- [db.md](docs/db.md)：讲结构和表职责
- [schema-reference.md](docs/schema-reference.md)：讲真实表和字段
- [status-codes.md](docs/status-codes.md)：讲数字编码
- 本文件：讲“业务流程是怎么走的”

## 总规则

1. 不要从单个页面反推完整流程。
2. 不要把老系统接口链直接当成新系统最终流程。
3. 如果数据库已经明确拆成配置层 / 动作层 / 结果层，流程也要按这三层理解。

## 当前已拍板的流程理解

### 后台权限

当前新系统后台权限流程：

- 后台账号：`admin_user`
- 角色：`admin_role`
- 权限：`admin_permission`
- 管理员绑定角色：`admin_user_role`
- 角色绑定权限：`admin_role_permission`

约束：

- 不再以 `user.role='admin'` 作为新系统后台权限模型
- 用户端账号和后台账号分离

### 店铺品牌授权

当前授权流程按两层理解：

1. 申请层
- `shop_brand_auth_apply`

2. 生效层
- `shop_brand_auth`

约束：

- 申请过程和生效关系不能混到一张表里理解
- 判断某店有没有品牌经营权限，先看 `shop_brand_auth`

### 优惠券

当前优惠券链固定按三层理解：

1. 模板层
- `coupon_template`

2. 发券动作层
- `coupon_grant_batch`

3. 用户券实例层
- `coupon`

约束：

- 不要把 `coupon_grant_batch` 当模板
- 不要把 `coupon` 当发券动作

### 排行榜

当前排行榜流程：

1. 应用层计算榜单规则和结果
2. 结果落到 `leaderboard_entry`
3. 页面读取 `leaderboard_entry` 展示

约束：

- `leaderboard_entry` 是结果表，不是规则表
- 用户昵称、头像等资料不再作为排行榜快照塞回榜单表

### 签到

当前签到链：

1. 奖励配置：`checkin_reward_config`
2. 用户签到结果：`checkin`

约束：

- 配置和结果不能混用
- 前台签到页若要显示第几天奖励什么，优先读配置表

### 邀请

当前邀请链：

1. 用户邀请码/邀请归属：`user.invite_code`、`user.invited_by`
2. 奖励配置：`invite_reward_config`

约束：

- 当前先不补独立邀请记录表
- 先按“关系 + 配置”理解邀请流程

### 寄售

当前寄售链：

1. 仓库资产：`physical_warehouse`
2. 寄售成交事实：`consign_trade`
3. 仓库日志：`warehouse_item_log`

约束：

- 仓库表负责资产状态
- `consign_trade` 负责成交事实
- 不要把成交行为继续揉回仓库主表

## 页面判断规则

### 如果页面在“配置某种东西”

优先判断是不是配置层页面，例如：

- 签到奖励配置
- 邀请奖励配置
- 优惠券模板
- 后台权限定义

### 如果页面在“展示一批结果”

优先判断是不是结果层页面，例如：

- 排行榜
- 发券结果
- 审核结果

### 如果页面在“处理一次动作”

优先判断是不是动作层，例如：

- 发券批次
- 寄售成交
- 举报处理

## Do Not Assume

- 不要把页面上的状态标签直接当流程定义
- 不要把旧系统 controller 名字直接当新系统业务边界
- 不要把某个字段存在就理解成某条流程完整闭环
- 不要把结果表当配置表
- 不要把配置表当流水表

## 什么时候需要补流程文档

遇到下面情况，就应该先补本文档再继续做：

- 页面在争论“这一步到底算申请还是生效”
- 页面在争论“这张表到底是配置还是结果”
- 同一个业务对象被多个线程按不同方式理解
- 页面和数据库结构能对上，但语义边界还是不清楚
